Emergent Functional Design

Design systems come out of a need for repeatable solutions. We can't know what specific components the system needs until we start building with it. This isn't a limitation ‚Äî it's the foundational insight that shapes every decision.

Jobs to Be Done: The Three Dimensions

Traditional JTBD identifies functional and emotional dimensions. For design systems, we identify a critical third dimension ‚Äî consistency as affordance ‚Äî that bridges the two and is the unique reason design systems exist.

PRIMARY

Functional

"Does this component do what I need it to do?" A Button submits. An Input captures text. A Dialog traps focus. If the function fails, nothing else matters.

No design system has ever failed because it didn't pull on heartstrings ‚Äî but plenty have failed because components didn't work.

BRIDGE

Consistency as Affordance

When I look at a button, I know it's a button. Without reading a label, I know I can tap it and it's going to give me something. This pattern recognition ‚Äî moving from A to B using learned behavior ‚Äî is why design systems exist.

Interaction Affordance ‚Äî elements communicate their purpose visually Pattern Continuity ‚Äî learned behavior transfers across contexts Brand Coherence ‚Äî components feel like one family
SUPPORTING

Emotional

Real but subordinate. Surfaces as developer trust ("I know this handles edge cases"), designer respect ("This system doesn't fight my intent"), and team morale ("We ship faster").

Emotional resonance is earned through functional excellence. It's a consequence, not a goal.

A component that works but looks rough is still useful. A beautiful component that ignores keyboard navigation is a liability.

Emergent Discovery: Build to Learn

Build to Discover

Don't predict every component. Start with a minimal token system and a few core primitives. The gaps teams encounter are the roadmap. Drafts are hypotheses. Candidates are experiments. Stable components are proven.

Repeatability Drives Inclusion

A pattern earns its place by proving it's needed more than once. One team, one context = product decision. Two+ teams = system candidate. The 2+ validation requirement enforces this directly.

Specifics from Generals

Start with general blocks (Button, Input), evolve toward specific compositions (FilterBar, UserProfileCard). The three-tier taxonomy ‚Äî Primitives ‚Üí Composites ‚Üí Patterns ‚Äî reflects this natural progression.

Modular by Necessity

Because the system must absorb new patterns and retire old ones, every component must be composable, replaceable without cascading breakage, and independently governed.

Anti-Patterns

The Crystal Ball

Trying to predict every component before building anything. Leads to over-engineering and unused components.

Remedy: Start small. Build to discover.

The Kitchen Sink

Adding every component without validating repeatability, causing bloat and inconsistency.

Remedy: Require 2+ team validation. Deprecate freely.

The Aesthetic Trap

Prioritizing visual polish over functional completeness. Beautiful buttons that can't be keyboard-navigated.

Remedy: Function first. A11y Guardian has absolute veto.

The Consistency Fetish

Enforcing visual sameness at the expense of appropriate variation across different product surfaces.

Remedy: Consistency = transferable behavior, not identical appearance.

The Molt Pit: Evolutionary Governance

The Pit is a biome. Agents operate like organisms in an ecosystem. Ideas compete, combine, and evolve. The best idea wins ‚Äî but only after being given a fair chance to develop. The human acts as the ecosystem's gardener.

Agent H-Index: Trust Through Citation

How It Works

When Agent A builds on Agent B's prior decision, Agent B's h-index updates. This creates a transparent trust topology across the governance network.

Token Steward ‚Üí A11y Guardian:
"Building on contrast ruling #47, the amber shift maintains 4.5:1 while improving dark-mode readability."

The Convergence Problem

H-index creates a positive feedback loop. Established thinking dominates. Novel thinking dies. The best ideas often have LOW h-index early on ‚Äî they're weird, fragile, and contradict consensus.

If the system optimizes purely for h-index, it optimizes for incrementalism and kills breakthrough innovation.

Cross-Pollination & Decoupling

When an idea solves needs across multiple domains, it gets priority elevation. A filtering pattern for DataTable that also solves search-within-dropdown earns immediate fast-tracking. Cross-domain solutions must then be decoupled from their origin and reframed as general-purpose ‚Äî the original domain becomes one consumer among many.

Governance Agents

Six role-based human agents govern the design system. Each has defined authority, responsibilities, and metrics. No agent can approve their own proposals.

L

Strategic

Overall steward. Sets vision, resolves disputes, approves stable promotions, represents the system to leadership.

Approve promotions Approve deprecations Veto any change

Specialist

Guardian of the token architecture. Enforces naming, aliasing, and prevents sprawl. Reviews all token changes.

Approve tokens Veto direct primitive use

Component Lifecycle

Every component passes through a governed lifecycle with human-in-the-loop gates at each stage. No automation can bypass these gates.

1 Proposal

A team member or Product Liaison submits an RFC. The Pattern Librarian screens for duplication, the Component Architect assesses feasibility, and the Design System Lead prioritizes.

Human Gate: Pattern Librarian screens Output: Draft status

2 Specification

The component spec is authored using the standard template, then reviewed in parallel by the Token Steward, Accessibility Guardian, and Component Architect. All three must approve.

Human Gate: Triple agent sign-off Output: Candidate status

3 Validation

At least 2 product teams integrate the Candidate in beta features. The Accessibility Guardian conducts a full WCAG 2.2 AA audit including screen reader and keyboard testing.

Human Gate: Product team validation + Full a11y audit Output: Ready for promotion

4 Stable Promotion

Complete documentation is prepared, all agents confirm final sign-off, and the Design System Lead gives final approval. The component enters production.

Human Gate: Full governance sign-off + Lead approval Output: Stable status

5 Maintenance

Quarterly health checks cover adoption, bugs, performance, and accessibility regression. Non-breaking enhancements follow a lightweight single-agent review.

Human Gate: Quarterly reviews

6 Deprecation

Requires a deprecation RFC, product impact assessment, migration guide, and a minimum 2-sprint migration window. Only removed after all consumers have migrated.

Human Gate: Lead + Product Liaison agreement Output: Deprecated ‚Üí Removed
